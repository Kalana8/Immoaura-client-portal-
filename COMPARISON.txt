❌ BEFORE (PROBLEMATIC - Race Condition)
═══════════════════════════════════════════════════════════════════════════════

generate_order_number() Function:
  
  SELECT COALESCE(MAX(CAST(SUBSTRING(order_number FROM 4) AS INTEGER)), 0) + 1
  FROM orders
  WHERE order_number ~ '^IM-[0-9]+$';

Problem Scenario - Two Users Create Orders Simultaneously:

  ┌─ TIME 0.001s ──┐    Current DB: [IM-000001]
  │  User A calls   │    
  │  generate_order │ → MAX() = 1, so returns IM-000002 ✓ (pending insert)
  │  _number()      │    
  └─────────────────┘    

  ┌─ TIME 0.002s ──┐    Current DB: Still [IM-000001]  ← Race condition!
  │  User B calls   │    
  │  generate_order │ → MAX() = 1, so ALSO returns IM-000002 ✓ (pending insert)
  │  _number()      │    
  └─────────────────┘    

  ┌─ TIME 0.003s ──┐    Insert IM-000002 → SUCCESS ✅
  │  User A inserts │    
  │  order          │    DB: [IM-000001, IM-000002]
  └─────────────────┘    

  ┌─ TIME 0.004s ──┐    Insert IM-000002 → FAILS ❌
  │  User B inserts │    ERROR: "duplicate key value violates unique constraint"
  │  order          │    
  └─────────────────┘    

Result: 50% of concurrent order creation attempts FAIL


✅ AFTER (FIXED - Using PostgreSQL Sequences)
═══════════════════════════════════════════════════════════════════════════════

generate_order_number() Function:

  next_number := nextval('order_number_seq');  ← ATOMIC!
  formatted_number := 'IM-' || LPAD(next_number::TEXT, 6, '0');

Fix Scenario - Two Users Create Orders Simultaneously:

  ┌─ TIME 0.001s ──┐    
  │  User A calls   │    
  │  nextval()      │ → Gets 2, returns IM-000002 ✓ (LOCKED from DB sequence)
  │                 │    
  └─────────────────┘    

  ┌─ TIME 0.001s ──┐    
  │  User B calls   │    
  │  nextval()      │ → Waits... then gets 3, returns IM-000003 ✓ (guaranteed unique)
  │  (same moment)  │    
  └─────────────────┘    

  ┌─ TIME 0.002s ──┐    Insert IM-000002 → SUCCESS ✅
  │  User A inserts │    DB: [IM-000001, IM-000002]
  └─────────────────┘    

  ┌─ TIME 0.002s ──┐    Insert IM-000003 → SUCCESS ✅
  │  User B inserts │    DB: [IM-000001, IM-000002, IM-000003]
  └─────────────────┘    

Result: 100% of concurrent order creation attempts SUCCEED! ✅

🔧 TECHNICAL DETAILS
═══════════════════════════════════════════════════════════════════════════════

PostgreSQL SEQUENCE (nextval):
  • Returns unique, incrementing integer values
  • ATOMIC - guaranteed no duplicates even under concurrent access
  • Specifically designed for this use case
  • Industry standard for auto-incrementing identifiers

MAX() Approach (old):
  • Not atomic - reads value, calculates, but no guarantee during insert
  • Susceptible to race conditions with concurrent transactions
  • Requires full table scan
  • Not suitable for high-concurrency scenarios
